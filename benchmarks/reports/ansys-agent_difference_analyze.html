1. 执行摘要

一句话结论：代码库在工程执行上超出了 Stage A/B 的规格，但核心数据资产（知识约束、陷阱、节点状态）仍为骨架状态；MCP 服务器与真实 AEDT 之间存在断裂；Stage C 范围已在 Stage B 基础未稳固前入侵。

该项目实现了令人印象深刻的基准测试框架、节点执行器和工作流引擎 — 但在原规格视为最关键差异化因素的方面却出现倒退：将详尽的 API 约束和领域知识注入 LLM 上下文。规格将"使用约束"描述为"LLM 最缺的" — 但这些字段目前在数据库中全部为空。

与此同时，21 个节点（含 8 个无实际逻辑的布局占位符）、DAG 工作流引擎、演化治理系统和 Web 演示 UI 已将项目推入 Stage C 领域，而 Stage B 的核心交付物（FastMCP 服务器 + 真实 AEDT）尚未完成集成。

2. 原始愿景回顾

项目核心宣言（摘自 设计规格 §12）：

本项目的核心不是让 LLM 记住 pyaedt，而是把 AEDT/HFSS 仿真流程拆成可约束、可检查、可复用、可演进的工程节点。

2.1 三阶段路线图

阶段目标关键交付物Gate
Stage A 证明轻量融合（知识 + 白名单 + 陷阱）优于裸 LLM 30 项基准测试 · A/B/C 组对比 · API 语义 DB · 7 个陷阱 · 离线评分 ✅ 已完成
Stage B 受控 MCP + 真实 AEDT 上的静态节点 FastMCP 服务器 · 8 个核心节点 · 会话管理器 · 执行队列 · AST 守护 · 审计日志 · 真实 AEDT 抽查 ⚠️ 部分
Stage C 图谱增强 · 节点演化 · DAG 运行时 GitNexusProvider · 完整 DAG · 可视化编辑器 · 节点版本治理 ⚠️ 过早

2.2 Go / No-Go 标准

进入 Stage B 必须满足（§7.5）：

  • Group C 比 Group B semantic pass 提升 ≥ 15%
  • Trap 捕获率 ≥ 80%
  • 至少 3 个任务类型真实 AEDT 抽检通过
  • 首批节点中至少 5 个达到 candidate-ready

2.3 关键约束

  • HFSS-only — 不含 Maxwell、Icepak、Mechanical、EDB、Optimetrics（§1.2）
  • 8 个核心节点 + 1 个待分解过渡节点（§5.1）
  • 正式路径不暴露裸 execute_script — 仅开发期可用（§6.1）
  • GitNexus 仅在 Stage C 中作为可插拔 Provider（§3.3）
  • Stage C 项目不得在 Stage A/B Go 条件通过前启动（§9.3）

4. 十大差距

GAP 1 · 知识层为空 极严重

规格要求：

约束字段（constraints_json、common_errors_json、common_traps_json）应包含真实使用约束 — "LLM 最缺的"数据。68 个 API 语义条目应有参数约束和示例引用。

当前状态：

api_semantics.seed.jsonl 的 68 条目中，全部深层字段为空：

"constraints_json": "[]"
"common_errors_json": "[]"
"common_traps_json": "[]"
"examples_ref_json": "[]"
"params_json": "[]"
"returns_json": "{}"
影响：该系统声称将 API 语义知识注入基准测试上下文，但约束存储是空的。Group B/C 的提升来自白名单约束（每节点限 3-5 个 API），而非知识检索。这相当于把核心价值主张掏空了。

4/7 缺失。基准测试中的陷阱任务（Trap_airbox_too_small 等）存在但引用的陷阱 ID 可能未定义。

GAP 3 · 节点目录超出范围 严重

规格要求：Stage A/B 的 8 个核心 HFSS 节点。不含 EDB、Maxwell、Icepak。

当前状态：nodes/catalog/ 包含 21 个节点，其中 8 个布局节点返回硬编码假数据（_layout_placeholder_node()）：

占位符节点实际行为
import_layout_file返回假 layout_path
select_layout_nets返回假 signal_nets
create_layout_cutout返回假 cutout_name
configure_layout_stackup返回假 stackup_name
create_layout_ports返回假 port_names
create_layout_setup返回假 setup_name
solve_layout返回假 solved_setup
create_layout_sparam_tdr_report返回假文件路径
违反规格：规格明确排除 EDB，且规定"Stage C 项目不得在 Stage A/B Go 条件通过前启动"（§9.3）。布局节点引入了 PCB/封装仿真范围，却没有对应的验证。

GAP 4 · MCP 服务器与真实 AEDT 断开 严重

规格要求：

MCP 服务器应维护持久化 pyaedt 实例，通过 .NET interop 与真实 AEDT 通信 — 这是 Stage B 的核心交付物。

当前状态：

mcp/server.py 硬编码 create_fake_kernel()，使用 FakeAedtAdapter。真实 AEDT 的 pyaedt_adapter.py 存在但未连接到 FastMCP 服务器。

唯一的真实 AEDT 执行路径是独立脚本 scripts/run_stage_c_real_workflow_smoke.py — 它完全绕过 MCP 协议和节点执行器。Stage B 的核心承诺"受控 MCP + 真实 AEDT"从未集成。

GAP 5 · 基准测试评估仅离线 中度

规格要求：4 层评估 — Syntax、API、Runtime（真实 AEDT 抽样）、Semantic（validation script 判断）。Trap 任务必须进入真实 AEDT 抽检（§7.2）。

当前状态：评分器仅检查 Syntax、Security 和 API pass。semantic_lite.py 只验证 Hfss 是否被导入 — 不是真正的语义验证。整个基准测试管道是纯离线运行。执行报告承认了这一点："当前 validation 检查的是结构性正确"，但未实现规格中的 Semantic Pass ≥ 70% 标准。

GAP 6 · 节点状态生命周期缺失 中度

规格要求：每个节点必须有 status: pseudo | candidate | candidate-ready | stable | deprecated 字段，并设定明确的过渡门控（§4.5、§5.4）。

当前状态：节点 YAML（如 create_substrate.yaml）没有 status 字段。演化模型定义了 ReviewStatus 枚举但未连接到节点目录。系统不知道哪些节点是"候选就绪"的，哪些是"稳定"的。节点目录是扁平的，没有治理。

GAP 7 · create_conductor_or_geometry_group 从未分解 中度

规格要求：该节点被标记为过渡节点，必须在满足触发条件时分解为 create_box_body、create_sheet_conductor、create_ground、create_patch、create_feed_geometry 等（§5.3）。触发条件：基准测试中 ≥3 次失败、API 白名单 > 8 个、postcheck 无法判断正确性。

当前状态：该节点仍为单体，接受通用 geometry 列表参数。分解逻辑从未启动。

GAP 8 · GitNexus 已放弃，无替代方案 中度

规格要求：GitNexus 是推荐的代码结构图谱工具，用于跟踪 API 调用链和生成节点的 api_scope（§3.8）。

当前状态：代码库中无 GitNexus 集成。official_retriever.py 存在但未被基准测试管道主动调用。从 pyaedt 源码自动生成 API 语义从未实现 — 68 个种子条目是手工编写的。

GAP 9 · 演化系统无实际效用 低

完整的演化包（evolution/）包含模型、方案提出器、挖掘器、评估器和策略 — 但它仅作为报告生成器运行（generate_node_evolution_report.py），而不修改节点目录。演化产生的是 HTML 报告，不是目录更新。

GAP 10 · 工作流案例不足 · 语义验证缺失 低

工作流案例：规格要求 7 个案例，实际仅 4 个。缺失：microstrip_line、rectangular_waveguide、coaxial_feed、cavity_resonator、simple_sparameter_export。

语义验证：规格中的 "Semantic Pass ≥ 70%" Go/No-Go 指标没有可操作的实现。当前 validation/ 模块仅检查对象是否存在、材料是否匹配 — 没有电磁物理正确性检查。

5. 架构亮点

尽管存在差距，以下工程成果值得认可：
模块评价
基准测试框架
benchmark/runner.py		 设计精良的 A/B/C 组对比，支持在线生成（OpenAI/Anthropic/file）和离线重放。Go/No-Go 计算、节点就绪度评估、HTML 报告管道均完备。
节点执行器
mcp/node_executor.py		 真正的工程级代码 — 实际上每个核心节点都被实现，具有正确的 AEDT API 调用、边界框计算、面选择、频率单位归一化、圆柱体几何推断。整个模块结构清晰，可直接部署。
工作流 DAG 模型
workflow/models.py		 结构清晰、可组合的 DAG 模型（Workflow → WorkflowNode → WorkflowEdge → WorkflowOutput），支持 $ref 参数引用、模板实例化、JSON 序列化，已做好产品化准备。
MCP 协议层
mcp/tools.py + mcp/server.py		 设计正确。6 个工具（create_session、release_session、list_available_nodes、describe_node、execute_node、get_model_info）遵循了规格的强制执行模式。dev_mode 开关控制 execute_script_restricted 访问。会话管理器、FIFO 队列、AST 守护、审计日志均就位。
假适配器
mcp/fake_aedt.py		 异常精细 — 以足够的保真度建模了真实 AEDT 对象状态（Face 对象含 id/center/area、box/cylinder/rectangle 几何类型、对象边界框计算、面选择逻辑、Sweep/Touchstone 导出）。使得无需真实 AEDT 即可运行完整的离线测试。
输入 Schema 验证
mcp/node_schemas.py		 每个节点有类型化输入 Schema，含 required/optional/defaults。sweep type 别名归一化、节点输出引用检测、频率单位解析均正确实现。

6. 关键风险矩阵

风险严重程度可能性说明
知识层为空 极严重 已发生 系统承诺的约束/陷阱注入完全未兑现。当前基准测试提升来源于白名单，一旦白名单效果边际递减，Group C 优势将消失。
MCP 与真实 AEDT 断开 极严重 已发生 Stage B 核心交付物缺失。如果用户试图将 MCP 服务器连接到外部 AI 客户端，所有 execute_node 调用都将落在假适配器上。
Stage C 范围入侵 严重 已发生 21 个节点（8 个占位符）、DAG、演化、Web Demo 在 Stage B 基础未稳固前构建。违反架构约束。
节点治理缺失 中度 已发生 无 status 字段、无稳定过渡、无目录修剪。GeometryGroup 从未分解。系统不能区分可部署节点和实验节点。
语义验证虚假 中度 已发生 semantic_lite_pass 仅检查 Hfss 导入。真实的电磁正确性从未验证。Semantic Pass ≥ 70% 的 Go 条件无法评估。

7. 建议的纠正路径

按优先级排序：

#行动影响工作量
P0 填充知识层。为所有 68 个 API 语义条目编写 constraints_json、common_errors_json、params_json 字段。每条目至少 2 条约束和 1 个常见错误。来源必须来自 pyaedt 源码或 AEDT 文档，禁止猜测。 立即提升基准测试 semantic_lite 分数；使知识检索成为真正的价值来源 2-3 天
P0 将 pyaedt_adapter 连接到 MCP 服务器。创建 create_real_kernel() 工厂方法，通过配置开关（环境变量或 config）在 fake/real 之间切换。确保 execute_node 在真实 AEDT 上可运行。 完成 Stage B 核心交付物；使 MCP 服务器可被外部 AI 客户端使用 1-2 天
P1 修剪节点目录。移除 8 个布局占位符（它们在所有代码路径中返回硬编码假数据）。为剩余 13 个节点添加 status 字段。将核心 8 个节点标记为 candidate，5 个后处理节点标记为 proposed。 恢复目录可维护性；启用节点治理 0.5 天
P1 完成 4 个缺失的陷阱 YAML。编写 waveport_no_background_contact、sweep_range_misses_target_frequency、material_or_unit_mismatch、boundary_assigned_to_wrong_object。连接到基准测试上下文构建器。 完成 Stage A 交付物；使陷阱任务可评分 1 天
P2 暂停 Stage C 范围扩展。直到 P0/P1 项完成且所有 Stage B Go 条件验证通过后，再考虑新的 DAG、演化或 Web 功能。 阻止进一步的架构债务 治理决策
