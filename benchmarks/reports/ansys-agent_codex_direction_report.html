1. 执行摘要

一句话结论：项目从“HFSS-only 受控节点仿真基座”演进到“workflow 产品化 demo 骨架”这个方向总体正确，但 Stage B 的对外执行入口、知识数据资产、节点治理还没有闭合；同时 Layout/BRD/EDB 方向已经明显超出初始范围，必须从主线中隔离。

我不同意把当前进展简单定义为“跑偏”。更准确的判断是：主技术路线选对了，但阶段门控没有跟上。Stage A/B 已经证明了两个重要事实：grounding 能明显提升 PyAEDT 生成效果；受控 node plan 比自由 Python 更可控、更可审计。Stage C 的 workflow、template、validator、demo server 是合理产品化延伸。

真正的问题不在于做了 Stage C，而在于 Stage C 的表层能力发展很快，底层的 Stage B contract 仍有缺口：FastMCP server 默认使用 fake adapter，API 语义库的约束字段基本为空，节点 catalog 没有稳定等级，Layout 节点以占位符形式进入了正式目录。

方向受控 workflow agent 是正确主线
风险Stage B contract 未闭合，Stage C 继续扩张
决策冻结扩张，补齐真实 MCP、知识层、catalog 治理

2. 项目方向判断

问题我的判断原因
项目是否偏离初始定义？ 部分偏离 HFSS 受控节点主线仍在，但 Layout/BRD/EDB 节点、demo tuning、import cutout 已超出初始 HFSS-only MVP。
Stage C 是否完全不该做？ 不应一刀切 workflow model、validator、executor、template 是受控节点路线的自然产品化，不应回滚；但继续扩功能前需要补 Stage B 缺口。
当前最大工程债是什么？ 入口 contract 不一致 真实 AEDT 能通过脚本跑受控 executor，但 FastMCP server 入口仍固定 fake，外部 agent 接入时拿不到真实执行能力。
当前最大产品风险是什么？ demo 叙事超过验证证据 报告和 UI 已经像产品，但 validation 仍以结构性检查为主，planner benchmark 和真实 smoke 覆盖都还偏小。

3. 与差距分析报告的一致点

3.1 知识层确实没有兑现初始价值主张

初始 spec 强调 API 语义、参数约束、common errors、common traps 是系统差异化基础。但当前 knowledge/api_semantics/api_semantics.seed.jsonl 的 68 条记录中，params_json、returns_json、constraints_json、common_errors_json、common_traps_json 基本为空结构。这样会让“知识检索”退化为 API 名称和签名提示，而不是工程约束注入。

判断：这是 P0 级缺口。如果不补，项目很难证明它的成功来自可维护知识资产，而不是 prompt、白名单和 benchmark 任务适配。

3.2 FastMCP server 与真实 AEDT 没有闭合

src/aedt_agent/mcp/server.py 中 create_server() 固定调用 create_fake_kernel()。真实 AEDT 路径存在于 PyaedtAdapter 和 smoke 脚本中，但没有通过 MCP server 暴露给外部 AI 客户端。

判断：这是 Stage B contract 的核心缺口。只要外部入口仍是 fake，Stage B 就不能算对外完成。

3.3 Layout/BRD 节点进入 catalog 是范围污染

初始 Stage A/B 明确 HFSS-only，不含 EDB。当前 nodes/catalog/ 已出现 import_layout_file、create_layout_cutout、create_layout_ports、solve_layout 等节点；其中一批在 NodeExecutor 中路由到 _layout_placeholder_node()，返回结构化假结果。

判断：这些节点不应出现在 stable/candidate 主 catalog。如果保留，必须标记为 experimental，并禁止默认 planner 选择。

4. 需要修正或补充的判断

差距报告中的说法我的修正执行含义
“真实 AEDT 执行完全绕过节点执行器” 不完全准确。真实 smoke 脚本没有走 MCP 协议，但确实使用了 SessionManager + PyaedtAdapter + NodeExecutor + WorkflowExecutor。 不需要重做真实执行链路；需要把已存在链路接到 FastMCP server。
“Stage C 过早，应暂停” 应暂停 Stage C 扩张，而不是回滚 Stage C 基础设施。workflow、validator、template 对产品化有价值。 保留已有 Stage C 骨架，冻结新功能，优先补 Stage B 门控。
“GitNexus 已放弃” 代码里仍有 GitNexus harness 配置、retriever 和 tool usage 分析；但它没有沉淀为可复用的 KnowledgeProvider 主链路。 短期不必强行重做 GitNexusProvider，先把已验证的检索结果固化进 API 语义库。
“语义验证虚假” 应区分 benchmark 的 semantic_lite 和 Stage C 的 model validation。后者不是物理正确性，但已经能检查对象、端口、边界、setup、sweep、report。 报告口径要改成“结构性 validation 已有，物理/结果 validation 不足”。

6. 建议路线

推荐策略：不要回滚 Stage C，也不要继续堆 demo。进入一个短周期的“Stage B/C contract stabilization”阶段，用 1-2 周把真实 MCP、知识层、catalog 状态、验证口径补齐。

6.1 主线应重新命名为 HFSS Controlled Workflow Agent

执行 agent 应把当前主线定义为：

自然语言 / 模板 -> workflow JSON -> validator -> controlled NodeExecutor -> AEDT -> model/result validation -> audit/report

这个定义既继承初始 spec 的节点化思想，也容纳 Stage C 已完成的 workflow/product 骨架。

6.2 Layout/BRD/EDB 应拆成 experimental track

Layout import、BRD cutout、PyEDB、HFSS 3D Layout 不是坏方向，但它们不是当前 HFSS MVP 的主线。建议建立独立 track：

  • 目录上标记为 experimental-layout。
  • 默认 demo/planner 不展示为稳定节点。
  • 真实实现和 placeholder 不能共用同一个稳定节点 ID。
  • 等有独立 spec、benchmark、real smoke 后再进入 candidate。

6.3 Validation 应分三层表述

层级当前状态下一步
结构性 validation 已有：对象、材料、端口、边界、setup、sweep、report 继续作为基础验收
结果级 validation 部分 demo 已导出 Touchstone / CSV 检查频点范围、S 参数文件可读、目标频率被覆盖、报告非空
电磁语义 validation 基本没有 只对少数模板加启发式规则，不要宣称完整物理正确

7. 给执行 agent 的任务优先级

优先级任务验收标准
P0 为 MCP server 增加 real/fake kernel 配置。复用现有 PyaedtAdapter、SessionManager、NodeExecutor，不要重写执行链路。 外部 FastMCP execute_node 可选择真实 AEDT；fake 仍为默认测试模式；有单元测试和手动 smoke 命令。
P0 给 node catalog 增加 lifecycle 字段：experimental、candidate、stable、deprecated。Layout 节点全部 experimental。 catalog JSON 输出包含 status；planner 默认只选择 candidate/stable；placeholder 节点不会被误报为稳定能力。
P1 补齐高频 API 语义字段。先覆盖 Stage B/C 当前真实用到的 API，不必一次性补满全部 68 条。 核心 20-30 个 API 至少有 params、constraints、common_errors、source_refs；context builder 能使用这些字段。
P1 补齐 common traps，并把 trap 与 benchmark/context/validation 连接起来。 7 个 trap YAML 齐全；trap 任务能在报告中显示被哪个规则捕获或未捕获。
P1 把 Stage C 报告口径改为“结构性验证 + 少量结果验证”，避免暗示完整电磁正确性。 README、stage progress report、HTML dashboard 中明确 validation 边界。
P2 为 microstrip、wave port、radiation airbox、dipole 增加结果级 validation。 Touchstone/CSV 文件存在且可解析；目标频率或 sweep range 被覆盖；基础曲线数据非空。

8. 建议的项目决策

建议保留：workflow model、validator、executor、template catalog、demo server、real smoke artifact、node evolution review。这些是受控 workflow agent 的正确产品化资产。
建议冻结：新的 demo 功能、更多 layout/BRD 节点、更多 tuning/optimization 能力。冻结不是删除，而是在 Stage B contract 补齐前不继续扩张。
建议立即整改：FastMCP real adapter、catalog status、layout experimental 隔离、知识库高频 API 约束、trap 规则。这些决定项目能否从“演示有效”进入“工程可信”。

最终判断

项目目前不是失败，也不是简单跑偏。它已经找到了正确的架构中心：LLM 不直接写可执行 PyAEDT，而是生成受控 workflow；系统用节点、schema、validator、真实 AEDT、audit 和 report 约束执行。

但这个中心需要被工程 contract 固定下来。执行 agent 接下来应少做展示层，多做闭环层：让同一套能力能通过 FastMCP 对外使用，能说明每个节点的稳定等级，能用真实知识约束模型输出，能用明确 validation 证明自己做到了什么、没做到什么。
