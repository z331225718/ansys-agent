# AEDT Agent Stage C.1 Demo/API Design

## Goal

Stage C.1 turns the current Stage C product skeleton into a small, demonstrable product loop. The target is not a full node editor yet. The target is a local web demo and API layer that let another team see the architecture, inspect nodes/templates, plan a workflow, validate it, run a controlled demo, and open the generated reports.

## Current Baseline

Stage C already has the core backend pieces:

- Node catalog with metadata/schema/version/UI hints.
- Workflow JSON model, validator, executor, templates.
- Fake adapter and real PyAEDT adapter.
- Inspector and model validation rules.
- Deterministic chat planner skeleton.
- Node evolution miner/proposer/evaluator.
- Chinese HTML reports and 3 real AEDT smoke artifacts.

Stage C.1 should reuse these pieces instead of building a second orchestration path.

## Users

Stage C.1 serves three audiences:

1. Non-Ansys or adjacent teams who need to understand what the agent does.
2. Simulation engineers who want to see node templates and execution artifacts before trusting the workflow model.
3. Developers who will later build the real UI, chat integration, and node evolution review process.

## Product Shape

Stage C.1 should provide a lightweight local web app backed by a small HTTP API.

The first screen should be the actual demo workspace, not a landing page. It should show:

- Current Stage C status metrics.
- Node catalog summary.
- Workflow template list.
- A selected workflow preview.
- Validation result.
- Run result and links to artifacts.
- Existing real AEDT smoke dashboard.
- Node evolution proposal review link.

The UI should be read/write enough to demonstrate the loop, but not a complete drag-and-drop graph editor.

## API Surface

The API should be intentionally small:

- `GET /api/status`
  - Returns repo/demo status, testable capabilities, and report links.
- `GET /api/nodes`
  - Returns serialized node catalog.
- `GET /api/templates`
  - Returns template summaries.
- `GET /api/templates/{template_id}`
  - Returns one full template and its workflow JSON.
- `POST /api/plan`
  - Input: user request plus optional parameters.
  - Output: planner result with selected template or generated workflow, missing information, assumptions, confidence, validation errors.
- `POST /api/validate`
  - Input: workflow JSON.
  - Output: workflow validation result.
- `POST /api/run`
  - Input: workflow JSON or template id plus parameters.
  - Output: fake-adapter run artifact paths and validation summary.
- `GET /api/reports`
  - Returns links to Stage C report, smoke dashboard, demo index, and node evolution review.

The API should use existing Python modules directly. It should not shell out to scripts except where a report generation script already represents the single supported artifact builder.

## Execution Policy

The web demo must default to the fake adapter. Real AEDT execution remains explicit and separate because it can be slow, machine-specific, and license-dependent.

For Stage C.1:

- `POST /api/run` runs fake adapter only.
- Real AEDT smoke is displayed from existing artifacts and can be regenerated by the existing CLI script.
- The UI can show the exact real smoke command, but should not launch AEDT from the browser in this milestone.

This keeps the demo responsive and avoids hiding AEDT license or process failures behind a web button.

## LLM Planner Boundary

Stage C.1 should add a replaceable planner boundary, but the first implementation can keep deterministic planning as the default.

Planner modes:

- `deterministic`: current local template/generation logic.
- `llm`: optional, configured through local config/env, never committed with secrets.

The LLM mode must obey the same rule as Stage C:

- LLM output is workflow JSON or a planner decision.
- LLM output is validated before execution.
- LLM never emits directly executable PyAEDT Python.
- Failed validation returns repair context, not execution.

If API credentials are missing, the system should fall back to deterministic mode or return a clear configuration error without breaking the demo.

## Config

Use a committed example config and an ignored local config:

- Commit: `config/demo_config.example.json`
- Ignore/use locally: `config/demo_config.local.json`

The example config should include blank fields for:

- planner provider
- model
- base_url
- api_key
- default adapter

No real key, base URL, token, or local absolute path should be committed.

## Reports and Artifacts

Stage C.1 should keep the current report strategy:

- Generated reports live under `benchmarks/reports/`.
- Demo run artifacts live under a small, selected path such as `benchmarks/runs/stage_c1_demo_latest/`.
- Large or repeated benchmark runs remain ignored.

The web app should link to report files instead of copying their contents into new places.

## Minimal UI

The UI should be practical and dense:

- Left column: nodes/templates/reports navigation.
- Main area: selected template/workflow preview and validation state.
- Right panel: run controls, status, artifact links.

Controls:

- Template selector.
- User request text area for planner demo.
- Parameter inputs for selected template defaults.
- Buttons for plan, validate, run fake demo, open report links.

Avoid full drag-and-drop node editing in Stage C.1. A static workflow preview is enough if it clearly shows nodes and edges.

## Node Evolution Thread

Stage C.1 should expose node evolution as a review view, not as an automatic mutation system.

The demo should show:

- Proposal count.
- Evidence count.
- Gate status.
- Required tests.
- Why proposals are blocked.

No proposal should be promoted to stable in Stage C.1 unless a separate reviewed implementation plan handles that candidate node end to end.

## Testing

Stage C.1 should be testable without AEDT:

- API tests using a local test client.
- UI/data contract tests for expected JSON shape.
- Script smoke tests for report generation.
- Existing full pytest suite must continue passing.

Real AEDT smoke remains opt-in and should not be part of normal test runs.

## Success Criteria

Stage C.1 is complete when:

- A local command starts the web demo.
- The demo can list nodes and templates.
- The demo can plan from a natural language request using deterministic mode.
- The demo can validate a workflow.
- The demo can run a fake-adapter workflow and show artifact links.
- The demo links to Stage C report, smoke dashboard, demo index, and node evolution review.
- API credentials are configurable but blank in committed files.
- Tests pass without real AEDT.

## Non-Goals

Stage C.1 will not build:

- Full drag-and-drop node editor.
- Browser-launched real AEDT execution.
- Multi-user job system.
- Cloud deployment.
- Physical/electromagnetic correctness proof.
- Automatic stable node release.
- General optimization agent.

These remain future phases after the demo/API boundary is stable.

## Recommended Implementation Order

1. Add config loader and example config.
2. Add API service layer around existing catalog/template/planner/validator/executor modules.
3. Add API tests.
4. Add lightweight web UI consuming the API.
5. Add demo start script.
6. Add report links and fake run artifact display.
7. Add documentation and final verification.
